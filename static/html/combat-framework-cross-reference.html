§II — Core 04

四个 核心框架

强烈建议掌握。不是"通识"——是经过厂商、学术、社区多轮淘洗后留下的最稳定四象限。

FRAMEWORK · 01 / 04

MDA 机制 · 动态 · 美感

Hunicke · LeBlanc · Zubek — AAAI 2004

学术界最常被引用的游戏设计框架之一。设计师自下而上看 M→D→A，玩家自上而下反推 A→D→M。

关键差异 · Key Difference

MDA 有"涌现"概念——Dynamics 是 Mechanics 跑起来后涌现的，不是设计的；本库四层模型把这一层做成了"主动设计的结构"。两者不冲突，但 MDA 提醒你：不是所有 Dynamics 都能被设计预见，需要 Playtest 反向归纳。

→ Aesthetics15_战斗体验设计目标 · 五维
→ Dynamics战斗设计四层模型 L2 + 内外循环
→ MechanicsL3 数值与公式层 + 系统架构
使用场景对外讲设计哲学 · 竞品逆向拆解 · 团队复盘从 A 层追溯到 M 层
FRAMEWORK · 02 / 04

3C Character · Camera · Control

Ubisoft 等 3D 动作 / 冒险开发语境

对 3D ARPG 来说，它适合作为立项 Prototype 的第一层检查骨架。Ubisoft 公开资料确认 3C 用于概念阶段的体验定义。

三轴定义 · Three Axes

Character · 角色"占多大空间、怎么移动、怎么转身"——碰撞体、骨架、locomotion。
Camera · 玩家"看到了什么、什么时候看不到"——跟随、锁定、构图、避障。
Control · 玩家"想做什么 → 角色做了什么"——按键、缓冲、响应延迟、状态转移。

→ Character3D 角色物理表达 + 04_动作系统
→ Camera战斗相机设计哲学 + 11_相机方案
→ Control10_输入系统 + 操控忠实度
使用场景立项期 Prototype 必须先锁 3C · 评估 3D ARPG 是否"立得住" · 程序 / TA 对接
FRAMEWORK · 03 / 04

Game Feel 三支柱 — Steve Swink

《Game Feel: A Game Designer's Guide to Virtual Sensation》2009

讨论"手感 / 打击感 / juice"时最值得优先掌握的框架之一。把"重量感、惯性、空间互动"列为独立支柱——不是反馈层的附属。

三支柱 · Three Pillars

Real-time Control 输入与角色行为之间的时间关系——延迟、响应、缓冲、取消。
Simulated Space 角色在世界中的物理存在感——碰撞、惯性、重量、地形。
Polish 表现层修饰——顿帧、震屏、闪白、特效、音效，俗称 "juice"。

→ Real-time Control操控忠实度 + 10_输入系统
→ Simulated Space战斗空间设计模型 · 距离/拓扑
→ Polish五层框架 L4 反馈层 + 12_表现
使用场景调试手感 · 技术美术讲 juice 配比 · 横向对比《只狼》《鬼泣》《战神》Polish 配方
FRAMEWORK · 04 / 04

OODA Loop 观察 · 解读 · 决策 · 执行

John Boyd · 美军飞行员决策循环

不是某厂专属方法论，而是迁移到战斗设计的对抗分析工具——解释玩家"从看到信息到采取行动"的全链路。

Observe → 看到敌人抬手 / 听到蓄力音 Orient → "这是横扫，不是突刺" Decide → "我应该侧闪而非格挡" Act → 按下闪避键 ↺ 结果产生新的 Observe

OODA 把 Orient（解读）单独拆出——决策慢不一定是反应慢，可能是"读不懂招式"。同样 0.6 秒预告，辨识度不同则 Orient 时间不同。

使用场景设计 Boss 招式信号（看到 vs 看懂）· 高难度可缩短 Observe 但不应缩短 Orient · 解释二周目背板效应
§III — Mid-Tier 08

八个 中级框架

值得知道——四个核心之外的标尺、模板与诊断工具。各自解一类具体问题。

3.1 — TEMPORAL NESTING

Core Loop 嵌套Multi-scale Loop Nesting

战斗不是单一循环，而是多时间尺度循环嵌套。Micro · Meso · Macro · Meta — 四层时间标尺。

MICRO 1–3 s · 普攻派生 / 闪避反击
MESO 30s–3min · 一波遭遇 / 单阶段
MACRO 5–30min · Boss 战 / 章节
META 1–50hr · 养成 / Build / 武器
本库映射　战斗内外循环 · 宏观难度节奏编排
3.2 — CURVE STACKING

Risk-Reward / Tension-Release风险-收益 / 紧张-释放

两条曲线叠加：风险-收益解决"为什么愿意冒险"，紧张-释放解决"压力如何随时间组织"。

★ 设计目标：让"高风险路径的期望收益 ≥ 保守路径"，否则无人选。

本库映射　战斗遭遇结构 · Boss 战节奏 · 宏观能量曲线
3.3 — TRIANGLE BALANCE

Combat Triangle三角平衡 / 克制循环

自《街霸》《魂》系列以来的老牌平衡论——格斗的"打 / 投 / 防"、魂的"攻 / 闪 / 格"、只狼的"打 / 格 / 跳"、对马的三剑型、鸣潮的元素相生相克。

差异化的最低成本工具：若三选一里有"绝对最优解"，差异化就崩了。

本库映射　克制关系设计模板 · 角色 / 武器 / 敌人差异化
3.4 — COMMITMENT COST

Action Economy动作承诺成本 · Frame Data

每个动作可拆为投入 · 命中 · 收回 · 改判四类成本——动作游戏调手感最常用的执行层语言。

STARTUP 起手 / 前摇
ACTIVE 有效帧 / 命中窗口
RECOVERY 收招 / 后摇
CANCEL 取消 / 缓冲
FRAME ADV 帧优势

动作游戏的"爽"不只是响应快，而是承诺成本可理解、可预期、可利用。

本库映射　帧数据统一模型 · 动作系统 · 取消机制
3.5 — SIGNAL LANGUAGE

Affordance / Signposting可感知性 · 可读性

来自 HCI / 交互设计——迁移到战斗后，主要解决"玩家能不能看出自己该做什么"。

OODA 解释玩家内部流程；Affordance / Signposting 解释系统如何帮助玩家完成 Observe 和 Orient。高难度可以惩罚错误决策，但不应让玩家输在"完全看不懂"。

本库映射　信号语言与可读性 · 12_表现 · 13_UI · 信息可读性
3.6 — FLOW STATE

Challenge-Skill / Flow挑战-技能匹配 · 难度曲线

Csikszentmihalyi 的 Flow 理论被借用为难度与学习曲线的校验工具。Boredom · Anxiety · Flow Channel · Mastery 四象限。

好的战斗难度不是"刚好不卡人"，而是让玩家失败后知道自己差在哪里。

本库映射　难度曲线与挑战-技能匹配 · Skill Atom / Skill Chain DAG
3.7 — MILLISECOND ANCHOR

Reaction Window Theory反应窗口 · 量化标尺

来自认知心理学与格斗社区交叉地带——所有"玩家需要在多少时间内响应"的定量标尺。

~200msSimple RT
~350msChoice RT
~300msRecognition
~50msPrediction

★ 200ms / 12 帧是动作游戏的"可反应性"基准。低于此阈值——你在考验反应力还是学习/预测？

本库映射　战斗信息识别 · 帧数据统一模型 · 操控忠实度
3.8 — COGNITIVE LIMIT

Hick's Law / Cognitive Load选择反应时 · 认知负荷

RT = a + b · log₂(n + 1) — 选项越多，决策越对数级变慢。工作记忆容量 3–5 块——超载时信号会被遗漏。

★ 设计原则：降低外在负荷（UI 干净）· 管理内在负荷（不要同时监控 6 种资源）· 保护相关负荷（保留招式学习空间）。

本库映射　信号语言 · 13_UI · 战斗资源经济与轮转

§Case — Field Study 02

实战 案例

框架不抽象——挂到具体作品的具体瞬间，才能验证它真有解释力。

CASE · OODA
魂系列 · 二周目背板效应

"为什么二周目
反而变简单了？"

一周目 · Boss 初见

Observe 看到 Boss 抬手——但不知道是什么招式。

Orient "这个起手姿势意味着什么？横扫？突刺？"——耗时最长，试错中学习。

Decide → Act "我不确定，先往后闪吧。"

二周目 · 背板后

Observe 看到 Boss 抬手。

Orient 瞬间匹配招式库——"是第三套连段的起手"——从推理模式切换为识别模式。

Decide → Act 决策已预编译，肌肉记忆执行。

这个差异不是数值变化，而是认知模型的彻底切换。Orient 是可训练的——好的敌人信号语言让玩家从"试错解读"平滑过渡到"快速识别"。
CASE · FLOW
艾尔登法环 · 大树守卫

"开场 30 秒后
就让你死的元教程"

初次遭遇 · 高挑战 × 低技能 → Anxiety

不知道骑马战斗、不知道可绕路、操作熟练度趋近于零——玩家处于 Anxiety 区，且失败原因不完全可理解。

设计师故意制造一次 Flow 违规，迫使玩家建立元认知："打不过就先绕路"。

中期回归 · 挑战适中 × 技能提升 → Flow Channel

骑马战斗已熟练 · 灵马、骨灰、更强武器 · 理解招式循环——玩家进入 Flow Channel，"每死一次都知道为什么"。

Flow ≠ 一直舒服。好的战斗设计不是让玩家永远在 Flow Channel——而是让偏离有意义，且偏离后有回到 Flow 的多条路径。
